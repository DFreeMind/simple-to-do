# 数据模型

## Root
- `schemaVersion: number`，当前为 `4`；表示前端状态结构版本。
- `groups: Group[]`
- `lists: List[]`
- `taskGroups: TaskGroup[]`，清单内任务分组。
- `tasks: Task[]`
- `trash: Task[]`
- `listTrash: ListTrash[]`
- `viewOrders: Record<string, string[]>`，普通任务视图的用户拖拽顺序；key 包括 `today`、`inbox`、`planned`、`important`、`list:<id>`、`search:<query>`。
- `settings: Settings`

## 存储位置
- `npm run dev` 使用 Tauri 开发模式，主数据存在 Tauri 应用数据目录：Windows 为 `%APPDATA%\cn.duqimeng.simpletodo\simpletodo.db`。
- 图片附件不写入数据库 BLOB，存放在同目录下的 `attachments/` 文件夹，数据库只保存附件元数据和相对路径。
- 只有单独运行 `npm run frontend:dev` 并在浏览器中打开页面时，才使用 `localStorage` 的 `simple-to-do:data` 作为前端调试兜底。

## Group
- `id: string`
- `name: string`
- `collapsed: boolean`
- `sortOrder: number`，侧栏分组手动排序权重。

## List
- `id: string`
- `name: string`
- `groupId: string | null`，所属侧栏分组ID，null 表示未分组。
- `color: string`
- `isSystem: boolean`
- `pinned: boolean`，是否置顶；置顶清单在侧栏"已固定"分组中置顶显示。
- `sortOrder: number`，侧栏清单手动排序权重。
- `viewMode: "list" | "group"`，任务显示模式，默认 `"list"`。列表模式平铺显示所有任务，分组模式按 TaskGroup 组织显示。

## ListTrash
- `id: string`，原清单 ID。
- `name: string`
- `groupId: string | null`
- `color: string`
- `isSystem: false`
- `sortOrder: number`
- `deleted: true`
- `deletedAt: string`
- `taskCount: number`，删除时清单内任务数量。

## TaskGroup（任务分组）
- `id: string`，分组唯一标识。
- `name: string`，分组名称，如"进行中"、"待审核"。
- `emoji: string`，可选的分组图标。
- `color: "auto" | "accent" | "blue" | "violet" | "amber" | "rose" | "green"`，分组背景色；`auto` 按主题和分组 ID 稳定生成。
- `listId: string`，所属清单ID，每个清单可有多个任务分组。
- `sortOrder: number`，分组在清单内的排序权重，数值越小越靠前。
- `collapsed: boolean`，分组模式下是否折叠，默认 `false`。
- `createdAt: string`，创建时间 ISO 格式。
- `updatedAt: string`，最后修改时间 ISO 格式。

## Task
- `id: string`
- `title: string`
- `description: string`，兼容旧纯文本描述。
- `descriptionHtml: string`，富文本备注 HTML；备注图片仅保存 `attachments/...` 相对路径，不保存 base64 预览数据。旧备注中的 base64 图片会在打开编辑器时导入附件目录并自动替换为路径引用。
- `completed: boolean`
- `completedAt: string | null`
- `deleted: boolean`
- `deletedAt: string | null`
- `deletedByListId: string | null`，因删除清单进入垃圾桶时记录来源清单。
- `pinned: boolean`
- `important: boolean`
- `myDayDate: string | null`，格式为 `YYYY-MM-DD`，表示加入某天的今日聚焦。
- `listId: string`
- `taskGroupId: string | null`，所属任务分组ID，null 表示未分组；仅在清单的 `viewMode="group"` 时使用。
- `dueDate: string | null`
- `reminderAt: string | null`
- `repeatRule: "daily" | "weekly" | "monthly" | "yearly" | string | null`
- `priority: 0 | 1 | 2 | 3`
- `tags: string[]`
- `subtasks: Subtask[]`
- `attachments: Attachment[]`
- `comments: Comment[]`，仅保留旧数据兼容，当前 UI 不展示评论入口。
- `editorMode: string`，兼容旧模式字段，当前 UI 不再使用模式切换。
- `createdAt: string`
- `updatedAt: string`

## Subtask
- `id: string`
- `title: string`
- `completed: boolean`
- `sortOrder: number`，子任务手动排序权重。

## Attachment
- `id: string`
- `kind: "image" | "file" | "unknown"`，附件类型。
- `mime: string`，MIME 类型。
- `originalName: string`，原始文件名，用于 UI 展示。
- `path: string`，旧字段兼容，保留原始路径或历史文件名。
- `relativePath: string`，相对 `attachments/` 的文件路径，例如 `images/2026/07/{sha256}.png`。
- `sha256: string`，文件内容 hash，用于去重。
- `sizeBytes: number`，文件大小。
- `width: number | null`，图片宽度。
- `height: number | null`，图片高度。
- `url: string`，运行时预览地址；外置附件保存时不写入该字段。
- `createdAt: string`
- `lastReferencedAt: string`

## Settings
- `theme: "mint" | "blue" | "violet" | "graphite"`
- `density: "comfortable" | "compact"`
- `detailOpen: boolean`
- `detailWidth: number`，详情面板宽度，范围 320-800px，默认 430px。
- `startView: "today" | "inbox" | "planned" | "important"`
- `completedVisible: boolean`
- `trashRetentionDays: number`，垃圾桶保留天数，默认 30 天。
- `soundEnabled: boolean`，操作音效总开关。
- `soundTaskEnabled: boolean`，任务操作音效开关。
- `soundListEnabled: boolean`，清单操作音效开关。
- `soundGroupEnabled: boolean`，分组操作音效开关。
- `reminderNotificationsEnabled: boolean`，系统提醒通知开关。
- `reminderSoundEnabled: boolean`，系统提醒声音开关；关闭后提醒尽量以静音通知展示。

## 兼容规则
- 当前开发阶段不迁移旧 `todo-data.json`。
- 前端 normalize 仍会为缺失字段补默认值，保证 SQLite 读取出的局部数据能稳定进入 UI。
- `comments` 和 `editorMode` 保留在前端结构中，避免后续迁移或导入时丢失历史字段。
- 删除任务先复制到 `trash`，永久删除才从 `tasks` 和 `trash` 移除。
- 删除清单先复制到 `listTrash`，清单下任务进入 `trash` 并标记 `deletedByListId`；恢复清单时一并恢复这些任务。
- 垃圾桶数据按 `settings.trashRetentionDays` 自动清理过期项。
- 前端 normalize 会为旧设置补齐提醒通知相关字段，默认开启系统提醒和提醒声音。
- 旧数据中 `taskGroups` 为空数组，`task.taskGroupId` 默认为 `null`，`list.viewMode` 默认为 `"list"`。

## 数据迁移

### 迁移处理器

数据版本迁移由专用模块 `src/utils/migrator.js` 负责，支持跨版本升级。

| 版本 | 迁移函数 | 变更内容 |
|------|----------|----------|
| v1 → v2 | （已有） | 基础结构升级 |
| v2 → v3 | `migrateV2ToV3()` | 添加 `taskGroups` 数组、`list.viewMode`、`task.taskGroupId` |

### 跨版本升级

当用户从旧版本（如 v1）直接升级到新版本（如 v3）时，迁移器会按顺序执行所有中间版本的迁移：

```
v1 → v2 → v3
```

每个迁移函数只依赖于前一个版本的数据结构，确保跨版本升级的安全性。

### 调用时机

应用启动加载数据时，`loadData()` 会：
1. 创建原始数据备份
2. 调用 `migrateData()` 自动检测并执行迁移
3. 调用 `validateData()` 验证迁移后的数据完整性
4. 验证通过后立即持久化迁移结果；验证或迁移失败时阻止写入，并提示用户恢复或升级应用

```js
import { migrateData, validateData, createBackup } from '@/utils/migrator'

// 1. 创建备份
const backup = createBackup(rawData)

// 2. 执行迁移
const migrated = migrateData(rawData)

// 3. 验证数据
const { valid, errors } = validateData(migrated)
if (!valid) {
  // 降级处理
}
```

### 错误处理

- 迁移前将原始状态备份到应用数据目录的 `backups/`；Web 模式保存到 localStorage。
- 缺少中间迁移、数据校验失败或读取到未来版本时，停止加载后的自动保存，避免覆盖原始数据。
- 数据版本高于当前应用支持范围时，要求升级应用，绝不降级写回。

### 扩展规范

新增版本迁移时：
1. 在 `migrations` 对象中添加迁移函数
2. 递增 `CURRENT_VERSION`
3. 在 `getSupportedVersions()` 中添加新版本号
4. 编写迁移逻辑（纯函数，可测试）
5. 添加对应的数据验证规则
