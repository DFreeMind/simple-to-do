# 数据库设计

## 状态
- 当前版本以应用数据目录中的 SQLite 数据库 `simpletodo.db` 保存任务状态。
- 当前开发阶段不迁移旧 `todo-data.json`。
- 图片附件按本文件的目录策略进入应用数据目录，数据库只保留附件元数据和相对路径。

## 文件布局
- 应用标识：`cn.duqimeng.simpletodo`
- 应用数据目录：由 Tauri `app_data_dir()` 决定。
- 主数据库：`simpletodo.db`
- 附件目录：`attachments/`
- 备份目录：`backups/`

## 当前开发态存储
- `npm run dev` 运行 Tauri 开发模式，当前写入应用数据目录下的 `simpletodo.db`。
- Windows 路径：`%APPDATA%\cn.duqimeng.simpletodo\simpletodo.db`。
- 附件路径：`%APPDATA%\cn.duqimeng.simpletodo\attachments\`，内部继续按 `images/YYYY/MM/{sha256}.{ext}` 分层。
- 开发态和安装包使用同一套 `app_data_dir()` 规则；如需隔离测试数据，应使用独立 identifier 或备份/清理该目录。

## 附件目录
- 图片：`attachments/images/YYYY/MM/{sha256}.{ext}`
- 普通文件：`attachments/files/YYYY/MM/{sha256}.{ext}`
- 未识别文件：`attachments/unknown/YYYY/MM/{sha256}.bin`

当前 MVP 只开放图片导入。目录预留普通文件、音频、视频和文档附件扩展。

## 附件命名规则
- 导入时计算文件内容 `sha256`。
- 扩展名优先由 MIME 推导，原始扩展名只作兜底。
- 同一内容 hash 已存在时不重复写入文件，只新增业务引用。
- UI 展示 `original_name`，不展示 hash 文件名。

## SQLite 表设计

## 扩展原则
- 业务表保留 `created_at`、`updated_at`、`deleted_at` 等生命周期字段，支持回收站、归档和发布后迁移。
- 多值关系使用关联表，例如标签、附件、视图排序，避免把高频扩展字段塞进大 JSON。
- 用户偏好和低频配置进入 `settings`，便于后续新增设置项。
- 发布后如果加入同步，可在核心表追加 `sync_id`、`device_id`、`dirty_at`、`last_synced_at`、`conflict_state`，不改变本地优先模型。
- 新增功能优先通过新表或可空字段扩展，不删除旧字段；所有结构变化必须写数据库迁移。

## 清理策略
- 删除任务附件或备注图片时只删除引用，不立即删除物理文件；回收站中的任务和清单仍保留其附件引用。
- 设置 > 数据中的“扫描本机存储”从已落盘 SQLite 引用和富文本 `description_html` 中收集路径，扫描 `attachments/` 后分别报告未引用附件与缺失文件引用。
- 未引用附件默认只能“移入清理站”，物理文件移动到应用数据目录的 `cleanup-bin/{批次}/`；用户可恢复或明确永久删除。发布版本不得静默删除用户附件。
- 扫描结果显示文件名、相对路径、大小、时间和图片缩略图。缺失文件引用只提示定位/修复，绝不自动删除任务内容。
- 删除任务或清单先进入回收站；超过 `trashRetentionDays` 后允许自动永久清理。
- 删除清单时，清单快照进入 `trash_lists`，清单下任务记录 `deleted_by_list_id` 并进入任务回收站；恢复清单时一并恢复这些任务。
