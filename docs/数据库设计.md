# 数据库设计

## 状态与边界

应用主存储是 Tauri 应用数据目录中的 SQLite 数据库 `simpletodo.db`。SQLite 表结构由 `src-tauri/src/main.rs` 初始化和迁移；本文件提供维护者可读的结构说明，代码是最终事实来源。

同一应用数据目录还包含：

```text
simpletodo.db                 主数据库
attachments/images/YYYY/MM/   图片附件，文件名为内容 sha256
profile/avatars/              自定义头像
backups/                      本机恢复点
cleanup-bin/                  附件清理站
```

当前产品只开放图片附件。`attachments/files/` 等路径不应在用户文档中承诺为可用能力。

## 表与职责

| 表 | 作用 |
| --- | --- |
| `meta` | 数据库结构版本与前端状态版本 |
| `groups` | 侧栏清单分组 |
| `lists` / `trash_lists` | 正常清单与删除清单快照 |
| `task_groups` | 清单内任务分组 |
| `tasks` | 任务主体与删除状态 |
| `subtasks` | 子任务与排序 |
| `task_tags` | 任务标签关系 |
| `attachments` / `task_attachments` | 附件元数据与任务引用 |
| `view_orders` | 视图内的手动排序 |
| `settings` | JSON 编码的应用偏好 |
| `clock_state` | 单行 JSON：专注预设、当前专注会话与专注历史；后续节律提醒数据也在此处扩展 |
| `profile` | 当前设备的昵称与头像元数据 |

## 关键约束

- ID 使用文本主键；排序使用 `sort_order` 数值；时间使用 ISO 文本。
- `tasks.list_id` 关联所属清单；`task_groups.list_id` 约束任务分组所属清单。前端还会校验任务与任务分组是否同属一个清单。
- 附件二进制不存入 SQLite。`attachments.relative_path` 指向应用数据目录下的文件，`sha256` 用于去重。
- 任务删除使用 `deleted` / `deleted_at`，删除清单使用 `trash_lists` 快照并标记相关任务的 `deleted_by_list_id`。
- `profile` 只保存本机资料，不保存密码、令牌或云端凭据；`account_id` 是未来映射预留字段，当前为空。
- `settings.value` 保存 JSON，因此新增设置项不需要新增列，但仍必须提供前端默认值和兼容处理。

## 结构升级与恢复

- Rust 以 `meta.databaseSchemaVersion` 记录表结构版本。打开已有数据库时先做只读健康检查；需要升级时在事务中执行。
- 数据库结构升级前使用 SQLite Online Backup API 创建一致性备份。前端状态升级与表结构升级是两件不同的事。
- 用户可见的恢复点会复制数据库、附件和头像；恢复前先创建“恢复前安全点”，失败时保留或回滚原数据。
- 不允许静默删除附件：扫描只找出未引用文件，用户明确将其移入清理站或永久删除后才改变物理文件。

## 维护规则

1. 修改表或索引时，同时更新 Rust 初始化、迁移、读写映射和本文件。
2. 先保证旧数据库可读，再考虑新字段写入；禁止通过删表或覆盖空库完成升级。
3. 对附件、头像、恢复点等文件操作使用相对路径并校验目录边界，不能接受任意绝对路径。
4. 发布前验证数据库读取、保存、附件导入、恢复点和垃圾桶恢复；有迁移时额外验证升级与失败保护。
5. 每次递增 `databaseSchemaVersion` 时，必须同步递增应用版本并发布支持该结构的安装包；开发版不得使用正式版数据目录。
