# 专注与节律产品开发任务

本文把产品规划拆成可交付、可测试、可验收的任务。状态以实际代码和验证结果为准；每完成一项，都要记录遗留问题并判断是否进入下一轮优化。

## 总体验收门槛

- 本机离线可完成专注、节律提醒、暂停/恢复、回顾和数据删除；旧数据库升级后任务数据不丢失。
- Windows 与 macOS 都提供等价的计时、通知、后台恢复和权限反馈；不能等价实现时，界面明确显示降级行为。
- 不记录键入内容、鼠标轨迹或前台应用名称；不作医疗效果承诺；用户始终可以暂停、延后或关闭提醒。
- 所有界面入口可用、可禁用或给出明确说明；任何上线功能通过 `npm run build`，涉及 Rust 改动还通过 `npm run tauri build` 或记录无法构建的原因。

## T1：时钟核心与本地数据基础

**状态：功能完成，等待跨场景验收**

### 交付内容

- [x] 数据模型 v10：专注预设、当前会话、会话历史和时钟设置；旧 JSON 自动补默认值。
- [x] 专注状态机：番茄、深度专注、自由计时；开始、暂停、继续、完成、提前结束、短/长休息与跳过休息。
- [x] 关联未完成任务或无任务专注；结束记录“完成 / 放弃 / 中断”及可选备注。
- [x] 当前会话、预设和历史写入既有 SQLite 保存路径，重启后能恢复正确剩余时间或暂停状态。
- [x] 初版专注工作台替换时钟占位页。
- [x] 补齐短休息/长休息阶段、跳过休息、编辑个人预设和结束备注。

### 测试

- [ ] 自动化：迁移 v9 → v10、空数据初始化、非法会话字段归一化（现有项目尚未配置单元测试脚本）。
- [ ] 手工：开始 1 分钟专注后刷新/重启，剩余时间按真实时间变化；暂停后重启，剩余时间不变。
- [ ] 手工：关联任务、解除关联、完成/放弃/中断均留下正确历史且不改变任务完成状态。
- [x] 构建：`npm run frontend:build`、`cargo check`。
- [ ] 构建：`npm run tauri build` 完整打包验证。

### 验收标准

1. 用户在 3 次点击内可开始默认专注；运行、暂停和结束状态都可见且互斥。
2. 时钟不依赖前端页面常驻；从后台、睡眠恢复或重启后，显示由保存的时间点计算的真实状态。
3. 保存失败时不清空会话或历史，并提示用户数据未保存。

### 优化复核

- 检查倒计时刷新是否造成不必要重渲染、不同模式是否容易理解、任务选择是否阻塞开始；必要时增加最近任务入口或缩短操作路径。

## T2：节律提醒引擎与模板

**状态：进行中**

### 交付内容

- [x] 提醒实体与首批模板：护眼、补水、站立/活动、久坐、眨眼、呼吸和自定义。
- [x] 间隔、固定时刻模型；工作日、工作时段、完成/延后/跳过/跳过当天和全局暂停。
- [x] 提醒中心、编辑表单与模板启停。连续活跃时长、静默时段和系统通知将在 T3 接入原生能力后完成。

### 测试与验收

- [ ] 每种触发器在边界时间、跨午夜、静默期、延后和跳过当天后的下一次触发均正确。
- [ ] 两条同时到点时按队列呈现且不丢失；关闭权限时保留提醒设置并说明未投递原因。
- [ ] 用户可以在不删除数据的情况下暂停全部提醒或恢复任意单条提醒。

### 优化复核

- 检查模板初始值是否过多、提示是否干扰专注；依据实际测试调整默认频率、文案与通知层级。

## T3：原生活跃时长、通知和后台体验

**状态：进行中**

### 交付内容

- [x] Windows 原生空闲时长 command；连续空闲 30 秒作为自然离席重置累计活跃时间。macOS/Web 明确降级为不支持该触发器。
- [x] 节律提醒的系统通知与应用内非阻塞提示；未授权时保留设置并静默回退。
- [x] 专注/休息截止由 Rust 后台调度；前台显示应用内完成弹层，最小化、隐藏或失焦时发送系统通知，并支持专用测试通知。
- [ ] 托盘/菜单栏快捷操作与节律提醒权限状态。
- [ ] 手动专注模式、全屏/演示/系统勿扰的可理解处理。

### 测试与验收

- [ ] Windows/macOS 分别验证最小化后台、锁屏、睡眠、恢复、通知拒绝、多显示器和全屏场景。
- [ ] 活跃时长不可用时明确改为墙上时间或禁用，绝不声称已检测键鼠活跃度。
- [ ] 不采集或持久化任何原始输入、轨迹或应用名称。

### 优化复核

- 测试通知是否在会议或专注时打断用户；评估需不需要调整延后限制、预告和浮层样式。

### 测试与验收

### 优化复核

- 以真实任务流走查“开始专注—被提醒—回顾”的连续体验，处理信息层级、空状态、性能和视觉一致性问题。

## 持续优化机制

每个任务完成后执行以下闭环：

1. 运行自动构建与相关手工场景，记录结果。
2. 对照验收标准逐项标记通过、失败或有条件通过。
3. 检查错误日志、保存/恢复路径、无障碍、跨平台差异和不必要的权限/数据收集。
4. 将影响核心闭环的缺陷回填到当前任务；可选增强进入后续任务，不以牺牲可靠性换取功能数量。
5. 更新本文、产品文档和使用指南，提交一个只包含该稳定功能的中文 commit。
